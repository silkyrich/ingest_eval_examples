## Summary

*Field limit: 80–3,000 characters. Current: ~1,847.*

INGEST_EVAL examples is a reference and teaching app that demonstrates how to transform data at index time in Splunk using INGEST_EVAL, CLONE_SOURCETYPE, REGEX, ADD_META, REPEAT_MATCH, and related mechanisms in props.conf and transforms.conf.

The app ships with eighteen worked examples, each including sample log data, a fully commented configuration, and a markdown writeup that explains the problem, the approach, the resulting events, and example SPL searches. Examples are grouped into five categories:

- Datetime problems: conflicting datetime formats on one sourcetype; stitching a date from a filename to a time from the event.
- Enrichment: dynamic extraction of indexed fields from attribute=value logs, event-length metadata for license back-billing, dropping unwanted INDEXED_CSV columns, enriching splunkd.log and splunkd_access.log, pre-processing JSON-Docker logs, and validated IPv4 extraction from unstructured text.
- Security: simple masking via double-ingestion, and advanced SHA1 masking with a reversible map event for privileged search.
- Platform extension: converting license_usage.log into metrics, exporting and importing events between Splunk instances, sharding data with splitByIndexKeys, and measuring ingestion volumes before onboarding new data.
- Complex forwarding: splitting forwarder output across multiple TCP streams to exceed the per-pipeline throughput ceiling, and fine-grained selective routing to multiple targets.

The app also includes an extended INGEST_EVAL reference that documents behavior absent from the public Splunk documentation, adapted from internal developer notes: _time versus $pd:_time$, multivalue handling with $mv:field$, the := assignment operator, explicit type coercion ([int], [float32-sf]), and guidance on high-cardinality string storage pitfalls.

This is a reference app. It is intended for Splunk administrators, consultants, and data engineers who need to solve real ingestion and enrichment problems, and who want a runnable, self-contained set of patterns they can adapt to their own deployments.

## Details

### Browsing the examples

After installation, open the "INGEST_EVAL examples" app from the Splunk navigation bar. The landing dashboard lists every example grouped by category, each with an icon, a short description, and a link through to the full writeup. From each example page you can:

- Read the problem statement and the proposed solution.
- View the props.conf and transforms.conf snippets involved, with inline comments explaining every non-obvious choice.
- See a sample event and the resulting indexed fields.
- Run an accompanying SPL search against the pre-populated sample data.

### Where the data lives

Examples index to one of three indexes created by the app: ingest_eval_examples_1, ingest_eval_examples_2, or shard_data_with_splitbyindexkeys. The mask_data_and_map, mask_and_clone, and selective_routing examples route cloned events across indexes to demonstrate CLONE_SOURCETYPE behavior.

### Adapting the patterns to your own data

The props.conf and transforms.conf stanzas are written as self-contained units with hyphen-namespaced stanza names (for example, mask_data_and_map-extract_password). You can copy a stanza into your own app, rename its sourcetype, and adjust the regex or INGEST_EVAL expression to match your data.

### Extended documentation

appserver/static/description/extended_documentation.md documents INGEST_EVAL behavior that is not covered in the public Splunk documentation, including PipelineData access, multivalue fields, type coercion, and the := assignment operator. It is recommended reading before adapting any of the patterns to production.

### Regenerating the sample data

The sample logs are generated by create_sample_data_sets.ipynb at the repository root. Running the notebook regenerates all sample files under appserver/static/sample_data/.

## Installation

### Requirements

- Splunk Enterprise 8.1 or later. INGEST_EVAL was introduced in 7.3, but several examples rely on features added in 8.x, including ADD_META behavior and the extended type coercion syntax.
- Write access to $SPLUNK_HOME/etc/apps/ on the indexer or heavy forwarder where index-time transforms are applied.

### Installing from Splunkbase

1. In Splunk Web, navigate to **Apps > Find More Apps**.
2. Search for "INGEST_EVAL examples" and click **Install**.
3. Restart Splunk when prompted.

### Installing manually

1. Download the `.tar.gz` package from the Splunkbase listing.
2. Extract it into `$SPLUNK_HOME/etc/apps/` so that the resulting directory is named exactly `ingest_eval_examples`. The bundled `inputs.conf` uses absolute paths of the form `$SPLUNK_HOME/etc/apps/ingest_eval_examples/appserver/static/sample_data/...` and will not find its sample data if the directory is renamed.
3. Restart Splunk:

   ```
   $SPLUNK_HOME/bin/splunk restart
   ```

### What the app creates on install

- Three event indexes: `ingest_eval_examples_1`, `ingest_eval_examples_2`, and `shard_data_with_splitbyindexkeys`.
- A `detritus` last-chance index used as the default destination for data that fails to route correctly.
- Monitor inputs on the bundled sample data under `appserver/static/sample_data/`.

### Where to install

This app must be installed on any Splunk instance that parses the data — that is, the indexer tier in a non-forwarder deployment or the heavy forwarder tier if you use heavy forwarders. The search-time dashboard assets also require the app on the search head. On a single-instance deployment, installing once is sufficient.

### Removing the app

To remove the app and its sample data:

```
$SPLUNK_HOME/bin/splunk remove app ingest_eval_examples
```

This does not delete the three indexes or the data already ingested into them. Remove those manually through **Settings > Indexes** if desired.

## Troubleshooting

### The app loads but the example dashboards are empty

The sample data is ingested by monitor inputs defined in `default/inputs.conf`. If no events appear:

- Confirm the app directory is named exactly `ingest_eval_examples`. The monitor paths are absolute and will not match otherwise.
- Check that the indexes `ingest_eval_examples_1` and `ingest_eval_examples_2` exist under **Settings > Indexes**. If they do not, verify that `default/indexes.conf` is being loaded by the same instance doing the indexing.
- Search `index=_internal sourcetype=splunkd source="*metrics.log" group=per_index_thruput` to confirm the indexes have received bytes.

### License usage metrics events do not appear in _metrics

The `metricify_license` example relies on `license_usage.log` events being generated locally and cloned via CLONE_SOURCETYPE. It will only produce output on an instance that itself emits `license_usage.log` (a license manager or a standalone instance). The transform filters on `type=Usage`, so only Usage rows are routed to `_metrics`.

### mask_data_and_map does not produce decoded search results

The search-time join depends on the cloned "map" event appearing at the same `_time` as the masked event. If your data source uses subsecond-precision timestamps and those subseconds differ between the two events, `group by _time` and `_raw` may fail to correlate them. Round timestamps or group by an indexed field instead.

### split_forwarding does not actually split to two forwarders

The example uses `_TCP_ROUTING` with a randomized suffix to demonstrate the technique but does not configure real target groups. To make it functional, add two `tcpout` groups named `split_forwarding_0` and `split_forwarding_1` to `outputs.conf` pointing at your real indexers.

### Indexed fields from auto_extract_indexed_fields are missing

The `ADD_META` + `REPEAT_MATCH` transform requires the regex to match with a leading whitespace character. Events where the first `attribute=value` pair sits at the very start of the line will not extract that first pair. Prepend a space or adjust the regex to `(?:^|\s)`.

### Warnings about a missing transform at startup

Earlier app versions referenced a transform that did not exist (`enrich_splunkd_access_log-move_host_to_meta`). This has been removed in the current release. If you see such warnings after upgrading, verify that the installed version matches the latest release on Splunkbase and that no stale copies remain under `local/`.
